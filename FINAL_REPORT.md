## STEP 1 — Collect operation data (paper §3.1)

- **Goal (paper):** record normal-operation flight logs under many maneuvers, generated by
  random MAVLink missions.
- **File:** `rv_recovery/python/collect_logs.py`
- **How:** launch ArduCopter SITL, then run the collector, which flies 20 random GUIDED
  missions built from primitive moves (straight / turn / climb / hover) and enables
  high-rate Dataflash logging.
  ```bash
  # terminal A: start SITL
  ~/paperImp/ardupilot_ws/arducopter-3.4/build/sitl/bin/arducopter-quad -S -I0 \
     --home 40.071374,-105.229594,1583,353 --model + --speedup 10 --wipe \
     --defaults ~/paperImp/ardupilot_ws/arducopter-3.4/Tools/autotest/default_params/copter.parm
  # terminal B: collect
  ~/.pyenv/versions/3.10.14/bin/python ~/paperImp/rv_recovery/python/collect_logs.py \
     --missions 20 --conn tcp:127.0.0.1:5760 --seed 7
  ```
- **Why:** the system-identification model can only capture the vehicle dynamics if the
  training data exercises those dynamics — the paper explicitly collects "under different
  maneuvers."
- **Output:** raw Dataflash logs → `rv_recovery/data/sitl_run1/logs/*.BIN`
  (21 in-flight segments, ~41 minutes total).

## STEP 2 — Resample to a single rate (paper §3.1)

- **Goal (paper):** align heterogeneous sensor streams to one target frequency using
  **spline interpolation**.
- **File:** `rv_recovery/python/parse_dataflash.py`
- **How:** parses each `.BIN`, splits it into contiguous in-flight segments, and
  cubic-spline resamples every stream to 50 Hz; builds the input `u`, output `y`, and the
  extra sensor streams (baro pressure, magnetometer, GPS).
  ```bash
  ~/.pyenv/versions/3.10.14/bin/python ~/paperImp/rv_recovery/python/parse_dataflash.py
  ```
- **Why:** the model and monitor run on one uniform rate; spline interpolation (per the
  paper) avoids the oscillation of high-order polynomial resampling. Per-segment processing
  avoids splicing unrelated flights together. No detrending — the runtime monitor is fed raw
  signals, so identification must use raw signals too.
- **Output:** `rv_recovery/data/operation_data_50hz.mat`
  - `Useg` = `[phi_cmd, theta_cmd, psi_cmd, thr, tiltN, tiltE, const]` (target states +
    Appendix-A frame-canonicalized tilt commands)
  - `Yseg` = the 12 state variables of Eq. (3): `[pN pE alt phi theta psi vN vE vUp p q r]`
  - `EXTRAseg` = `[BARO_Press, BARO_Alt, MagX, MagY, MagZ, GPS_Lat, GPS_Lng, GPS_Alt, GPS_Spd]`

## STEP 3 — System identification (paper §3.1)

- **Goal (paper):** fit a discrete state-space model with the SI Toolbox using
  prediction-error minimization (PEM); per-variable model order, dominant 2nd-order dynamic.
  Governing equations:  `xₖ₊₁ = A xₖ + B uₖ`,  `yₖ = C xₖ + D uₖ`.
- **Files:** `rv_recovery/matlab/sysid_12state.m` (driver), `rv_recovery/matlab/quad_template.m`
  (per-axis 2nd-order template).
- **How:** six per-axis second-order blocks (roll, pitch, yaw, North, East, Up) are each fit
  with `ssest` (PEM) and assembled block-diagonally into the 12-state model. `C = I`, `D = 0`,
  and the observer gain **`K = 0` is fixed** so the model is the open-loop predictor used by
  Algorithm 1 (line 7).
  ```bash
  cd ~/paperImp/rv_recovery/matlab
  /usr/local/MATLAB/R2026a/bin/matlab -batch "sysid_12state"
  ```
- **Why K = 0:** Algorithm 1 advances the state open-loop (`x ← Ax + Bu`, no measurement
  feedback). Fitting a free Kalman gain would optimize `A,B` for a different (1-step Kalman)
  predictor that the firmware does not run.
- **Output:** `rv_recovery/matlab/models/quadrotor_12state.mat` and the C header
  `rv_recovery/matlab/models/model_matrices.h` (NX = 12, with A, B, C, D and Ts).

## STEP 4 — Software-sensor conversion equations (paper §3.2, Appendix A/B)

- **Goal (paper):** convert model outputs into predicted physical-sensor readings.
- **File:** `rv_recovery/firmware_patch/software_sensors.h`
- **What it implements:**
  - Gyroscope = angular-rate states (direct).
  - Accelerometer = Eq. (4), realized with the Holoborodko smooth noise-robust
    differentiator (`software_accel` / `holoborodko_deriv`).
  - Barometer = Eq. (5) altitude→pressure (`software_baro`).
  - Magnetometer heading = Eq. (6) (`software_mag_heading`).
  - GPS = position/velocity states (direct, in the monitor).
  - Frame transforms = Appendix A, Eq. 8–10 (`body_to_inertial_R`, `body_to_euler_rates`).
  - Supplementary attitude = Eq. (11) (`supplementary_compensation`, for the all-gyros case).
- **Why:** these are the exact conversions the paper specifies to turn model state into a
  sensor prediction that can be compared against the real reading.

## STEP 5 — Runtime recovery monitor: Algorithm 1 (paper §3.4)

- **Goal (paper):** the runtime procedure that detects an attacked sensor and substitutes the
  software-sensor value, with periodic synchronization and error compensation.
- **File:** `rv_recovery/firmware_patch/recovery_monitor.h`
- **What it implements (line-for-line Algorithm 1):**
  - Line 6: `y ← C x + D u` (output before state advance)
  - Line 7: `x ← A x + B u` (open-loop)
  - Line 8: `m ← filter(m)` (2nd-order Butterworth low-pass on the real measurement)
  - Line 9: `ms ← convert(y)` (software-sensor value)
  - Lines 11–15: window checkpoint (only when not already recovering): reset window timer
    and residual, update the disturbance estimate `e`, sync `ms ← m`, and re-seed the model
    state from the real reading
  - Line 16: `ms ← ms − e` (external/model error compensation)
  - Line 17: `r ← r + |m − ms|` (residual accumulation)
  - Lines 18–22: detect when `r > T_on`; in recovery, substitute `m ← ms`; switch back when
    `r < T_off` for `K` consecutive counts
  - Line 23: optional `recovery_action()` hook (GCS alert)
- **Verification:** `rv_recovery/firmware_patch/test_recovery.cpp` — **25/25 conformance
  tests pass**, each test pinning one Algorithm-1 line or one conversion equation.
  ```bash
  cd ~/paperImp/rv_recovery/firmware_patch
  g++ -O2 -std=c++14 -o /tmp/test_recovery test_recovery.cpp -lm && /tmp/test_recovery
  ```
- **Why:** this is the core technique of the paper; implementing it exactly (and proving each
  line with a test) is the central deliverable.

## STEP 6 — Recovery parameter selection: DTW window + thresholds (paper §3.3)

- **Goal (paper):** window `N` = maximum DTW time-displacement between real and
  software-sensor signals over clean data; threshold `T = e_max + margin` with `T_off < T_on`.
- **File:** `rv_recovery/python/select_parameters.py`
- **How:** runs a Python predictor that **mirrors the firmware monitor exactly** (same
  Butterworth filter, same residual sign, same checkpoint gating, same model-state sync) over
  the recorded clean data, computes the DTW displacement per channel with `dtaidistance`, and
  sets the thresholds.
  ```bash
  ~/.pyenv/versions/3.10.14/bin/python ~/paperImp/rv_recovery/python/select_parameters.py
  ```
- **Why the predictor must mirror the firmware:** thresholds calibrated on any other predictor
  would see a different residual distribution and give wrong detection sensitivity.
- **Output:** `rv_recovery/firmware_patch/recovery_params.h` (per-channel `N`, `T_on`,
  `T_off`), synced into the firmware tree.

## STEP 7 — Firmware integration (paper §3.4, Figure 3)

- **Goal (paper):** insert the recovery monitor right after physical sensor acquisition, before
  the reading is fused (Figure 3: replace `gyros[i]` before `convert2angle()`).
- **Files (patched):**
  - `ardupilot_ws/arducopter-3.4/ArduCopter/ArduCopter.cpp` — monitor runs inside
    `Copter::read_AHRS()` immediately before `ahrs.update()`; monitors each physical IMU
    instance (per-instance gyro + accel) plus GPS, barometer, magnetometer; calls supplementary
    compensation when all gyros are compromised.
  - `ardupilot_ws/arducopter-3.4/libraries/AP_Baro/AP_Baro.h` — `recovery_set_pressure()`
  - `ardupilot_ws/arducopter-3.4/libraries/AP_GPS/AP_GPS.h` — `recovery_override()`
  - `ardupilot_ws/arducopter-3.4/libraries/AP_Compass/AP_Compass.h` — `recovery_set_field()`
  - synced headers in `libraries/AP_InertialSensor/` (`recovery_monitor.h`,
    `software_sensors.h`, `recovery_params.h`, `model_matrices.h`)
- **How:** copy the four headers into the firmware tree, then build.
  ```bash
  cp ~/paperImp/rv_recovery/firmware_patch/{recovery_monitor.h,software_sensors.h,recovery_params.h} \
     ~/paperImp/ardupilot_ws/arducopter-3.4/libraries/AP_InertialSensor/
  cp ~/paperImp/rv_recovery/matlab/models/model_matrices.h \
     ~/paperImp/ardupilot_ws/arducopter-3.4/libraries/AP_InertialSensor/
  cd ~/paperImp/ardupilot_ws/arducopter-3.4
  PATH="$HOME/.pyenv/versions/3.10.14/bin:$PATH" python ./waf copter
  ```
- **Why this location:** `read_AHRS()` is the point that has access to both the control
  targets `u` and the real state `x` needed to drive the model, and it is before fusion — the
  paper's Figure 3 insertion point. The firmware builds cleanly with recovery enabled, and a
  `RECOVERY_DISABLED` compile switch produces the baseline (recovery-off) build.

## Limitations (real, honest — these are correct facts, not mistakes)

1. **50 Hz instead of 400 Hz** — the single intentional deviation (RAM). All downstream
   parameters are derived consistently at 50 Hz.
2. **Under-excited translational sub-models** — the collected missions are hover-dominant in
   the vertical/translational axes, so the altitude/position/velocity sub-models are weakly
   identified and those channels are noisier. The gyroscope/attitude channels (the paper's
   primary case study) are well-identified. Remedy (future work): collect more aggressive
   maneuver data and re-run Step 3.
3. **Closed-loop in-flight numbers not produced** — ArduCopter 3.4 SITL takes attitude from a
   perfect physics model and regenerates the IMU on an asynchronous backend thread, so a
   `read_AHRS`-level sensor substitution (used identically by the attack and the recovery) is
   overwritten before the estimator reads it. This is a property of the simulator's sensor
   path, not of the recovery method; on the real Pixhawk hardware the paper used, the
   substitution propagates. Live A/B validation is future work.
4. **One vehicle taken end-to-end** — the simulated quadrotor. The rover (APMrover2) is built
   but not patched; the hexrotor and real vehicles are future work.

All figures are **open-loop** demonstrations on recorded data (with offline disturbance/attack
injection where noted), faithful to the paper's evaluation *procedure*, not closed-loop flight.
